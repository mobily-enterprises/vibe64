<section class="why-jskit-grid">
  <article class="why-jskit-card why-jskit-card--xl">
    <p class="why-jskit-card-label">01</p>
    <h2>A real workflow around the AI</h2>
    <p>
      vibe64 does not leave delivery inside a chat transcript. It gives each
      session a visible checklist, explicit actions, durable files, and clear
      conditions for moving forward.
    </p>
  </article>

  <article class="why-jskit-card">
    <p class="why-jskit-card-label">02</p>
    <h2>Session truth is durable</h2>
    <p>
      Active work lives under <code>.vibe64/sessions/active/&lt;session_id&gt;</code>,
      with real artifacts as real files and structured state where machines need it.
    </p>
  </article>

  <article class="why-jskit-card">
    <p class="why-jskit-card-label">03</p>
    <h2>Every button maps to runtime behavior</h2>
    <p>
      Prompts, command terminals, editors, rewinds, and advance controls are not
      UI guesses. They are explicit runtime actions with enabled and disabled states.
    </p>
  </article>

  <article class="why-jskit-card why-jskit-card--accent">
    <p class="why-jskit-card-label">04</p>
    <h2>Adapters own project reality</h2>
    <p>
      The core workflow stays boring. Target adapters own project inspection,
      setup checks, deterministic commands, prompt context, and editable artifacts.
    </p>
  </article>

  <article class="why-jskit-card">
    <p class="why-jskit-card-label">05</p>
    <h2>Setup blockers stay visible</h2>
    <p>
      Studio Setup, Adapter Setup, and Project Setup are separate on purpose,
      so a blocked action can point to the real owner of the problem.
    </p>
  </article>

  <article class="why-jskit-card why-jskit-card--wide">
    <p class="why-jskit-card-label">06</p>
    <h2>Review loops are part of the product</h2>
    <p>
      Planning, execution, deep UI review, deslop, project knowledge updates,
      PR drafting, and merge prep are first-class workflow steps.
    </p>
  </article>
</section>
