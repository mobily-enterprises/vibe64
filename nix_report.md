# Nix Runtime Plan From The User Perspective

This report describes the intended Vibe64 experience after Docker is removed
from generated software deployment and development. The goal is simple:

> A Vibe64 project declares the runtimes it needs, Vibe64 offers only supported
> versions, and Nix provides those exact binaries on Linux without containers.

Nix is the package substrate. Vibe64 remains the product runtime manager.

## Execution Status - 2026-07-06

This plan is implemented for the first useful no-container JSKIT runtime slice
and has been smoke-tested locally with Nix, managed MySQL, and Codex app-server
startup. The broader runtime matrix and deployment criteria are still not all
done.

### Still Incomplete

- The runtime env resolver is not yet a single central execution substrate for
  every Doctor repair, PTY command terminal, launch target, target script, and
  deployment phase. The current slice covers the shared Node command helpers and
  JSKIT checks first.
- Postgres, alternate Node versions, PHP 8.4, MySQL 8.4, and
  `laravel-installer` are catalog/planning scope only. Laravel PHP 8.3 and
  Composer command execution is wired, but a full Laravel product smoke has not
  been run yet.
- Runtime selection UI now covers adapter-owned choices that already have a
  runtime package contract, including JSKIT/Laravel database runtime choices.
  It is not yet a generic arbitrary package/version picker, and alternate Node,
  PHP, MySQL, or Postgres versions are still catalog/planning scope.
- Next.js, generic Node web, and Vinext now declare Node runtime requirements,
  but their full Doctor/preview/deployment paths still need broader
  end-to-end product smokes under the project runtime contract.
- Adapter-owned hosted config is not new Nix scope. The remaining Nix work is
  to make runtime/service selections feed the existing adapter-owned config and
  Runtime Config paths consistently, without introducing generic Vibe64 app
  config forms.
- Online deployment runner preserves runtime-wrapped adapter publish commands,
  but real publish/rollback smoke has not been run. Host setup and Vibe64
  daemon startup are Nix-aware; app publish now depends on adapters providing
  the runtime-aware publish plan consistently.
- Local API and browser-driven product smoke now covers JSKIT runtime
  selection, runtime lock writing, managed MySQL startup, project shell render,
  session creation, workspace preparation, the session Config tab, and Codex
  app-server startup. Full seed workflow completion and cleanup are not yet
  recorded because the user is actively using the smoke instance.
- Codex app-server currently logs one Linux sandbox/user-namespace warning on
  startup. The host sysctls and `bwrap` binaries are present, Vibe64 records
  Codex app-server readiness, and the assistant is answering in the UI, so this
  is a follow-up Doctor/reporting item unless a command execution path proves it
  is fatal.
- Online local HTTP smoke passed. Browser-driven Online workflows have not been
  run after these changes.
- No VPS deploy was run, and none should be run until explicitly requested.

## Core Rule

Vibe64 should keep the same ownership model it already uses for Runtime Config:

```text
Vibe64 owns supported versions and resolved runtime values.
Adapters describe the technology shape.
Generated files are compatibility output.
Nix supplies reproducible binaries.
Doctor proves the host and project match the contract.
```

For example, the JSKIT adapter can say "this project needs Node and may need a
MySQL-compatible database." Vibe64 decides which Node and MySQL versions are
supported. The project stores a selection from that approved set. Nix realizes
the selected binaries. Doctor verifies the result.

## First Start

### What The User Sees

On first start, Vibe64 opens Setup/Doctor before the user creates or runs a
project.

Doctor shows a "Vibe64 runtime" section with checks like:

```text
Linux host
Nix installed
Vibe64 base toolchain realized
git
ripgrep
Playwright
Codex
opencode
bubblewrap
```

The user should not be asked to install MySQL, Node, PHP, Composer, or database
clients manually. If Nix is missing, Doctor blocks and provides a guided install
or host setup command. It should not silently install Nix without an explicit
user or host-admin action.

For Vibe64 Online, host setup should install and verify Nix as part of the
owner/workspace bootstrap. For CLI/local users, the CLI should run the same
Doctor checks and present the same fix path.

### What Happens Under The Hood

Vibe64 has a base runtime catalog. It contains hard Vibe64 requirements such
as:

```text
git
ripgrep
playwright
codex
opencode
bubblewrap
node used by Vibe64 tooling
```

Doctor does not trust ambient `/usr/bin/git`, `/usr/bin/node`, or
`/usr/local/bin/codex` as the final answer. It realizes the Vibe64 base
toolchain through Nix, checks the store-backed binaries, then runs smoke tests.

Some checks are binary checks:

```text
git --version
rg --version
codex --version
```

Some checks are capability checks:

```text
bwrap can actually start with the host kernel settings
Playwright can launch Chromium
Codex/opencode auth state is usable
```

This matters because Nix can provide the `bwrap` binary, but the host kernel can
still prevent user namespaces. Nix can provide Playwright and browser packages,
but Doctor still needs to launch a browser.

The long-term Online systemd service should start Vibe64 through a Vibe64
runtime wrapper, not through a random host `node`. The current service starts
with `/usr/bin/env node`; the Nix version should use the pinned base runtime
after a minimal shell/bootstrap step.

## Creating The First JSKIT Project

### What The UI Does

The current project setup already starts with app type and technology:

```text
App type -> Technology -> Project config
```

For a first JSKIT project, the user chooses the app type, then JSKIT as the
technology. Vibe64 saves the project type the same way it does now:

```text
<project>/.vibe64/project_type
```

Adapter config continues to live in source-owned Vibe64 config:

```text
<project>/.vibe64/config/*
```

For JSKIT, the existing database choice is already an adapter config concept:

```text
jskit_database_runtime = none | mysql | postgres
```

The new runtime UI should appear immediately after technology selection, or as
part of the same Configure/Setup flow. It should show only the versions Vibe64
supports for that adapter.

Example JSKIT defaults:

```text
Node: 22
Database runtime: MySQL-compatible
Database version: MySQL 8.0
```

If the JSKIT project chooses no database, no database service is prepared. If it
chooses a MySQL-compatible database, Vibe64 prepares a pinned MySQL runtime,
database name, app user, and password.

The user is choosing from Vibe64-approved options. They are not typing arbitrary
versions such as `mysql 5.7.12`.

### Where The Settings Live

The selected project type and adapter choices should remain source-owned
project config:

```text
<project>/.vibe64/
  project_type
  config/
    jskit_database_runtime
```

Runtime package/version choices are Vibe64 runtime choices, not adapter-owned
app settings. They should be stored in a Vibe64-owned runtime selection/lock
record in the source-owned control area, for example:

```text
<project>/.vibe64/runtime.lock.json
```

That lock records the exact Vibe64 catalog entry and Nix resolution used for
the project. It should be committed with the project because it is part of the
runtime contract.

Realized binaries, service data, sockets, ports, logs, and secrets do not live
in source. They live under the existing Vibe64 runtime roots:

```text
local:
  ~/.local/state/vibe64/projects/<slug>-<hash>/runtime/
  ~/.local/state/vibe64/projects/<slug>-<hash>/runtime-config/

online:
  /var/lib/vibe64/<owner>/services/
  /var/lib/vibe64/<owner>/projects/<project>/
```

This keeps portable project intent in Git and machine-specific state out of
Git.

### What Happens Under The Hood

The adapter exposes a runtime contract:

```text
JSKIT:
  tools:
    node
    npm/corepack
  services:
    database, when selected
```

Vibe64 intersects that contract with the global runtime catalog:

```text
node:
  allowed: 20, 22
  default: 22

mysql:
  allowed: 8.0, 8.4
  default: 8.0

postgres:
  allowed: 16, 17
  default: 17
```

Then Vibe64 resolves the selected entries to Nix packages and records the
result in the project runtime lock. Commands do not run through ambient PATH.
They run through a Vibe64 runtime environment that puts the resolved Nix
binaries first.

The existing command surfaces should all consume the same runtime environment:

```text
Doctor checks
Doctor repair terminals
Project setup commands
Workflow command terminals
Target scripts
Preview launch targets
Deployment publish phases
Release service start scripts
```

Adapters should not manually call `nix develop`. They should continue to ask
for `node`, `npm`, `php`, `composer`, `mysql`, or `psql` through Vibe64's
runtime-aware command layer. Doctor can still report that the result came from
Nix:

```text
Node 22.11.0 from Vibe64 Nix runtime
MySQL 8.0.37 from Vibe64 Nix runtime
```

## Runtime Services

### What The User Sees

When a project chooses MySQL, the user should see a managed service status:

```text
MySQL 8.0
Initialized
Database created
App user valid
Generated env synced
```

The user should not create the database, user, or password manually.

The Env screen continues to show generated files and values. Vibe64-owned
database values are read-only. User-owned values remain editable.

### What Happens Under The Hood

Vibe64 initializes the service from the pinned Nix MySQL binary:

```text
runtime service root/
  mysql/
    data/
    run/
    logs/
    my.cnf
    secrets.json
```

On first init, Vibe64:

1. Creates a project-local data directory.
2. Starts the pinned MySQL server on a Vibe64-owned socket or localhost port.
3. Generates and stores an admin credential in Vibe64-owned secret storage.
4. Creates the app database.
5. Creates the app user.
6. Grants that user only the app database privileges.
7. Writes runtime config records for the adapter.
8. Materializes generated `.env` files when the adapter requires them.
9. Runs a connection check as the app user.

Later runs are idempotent. Vibe64 verifies existing state instead of blindly
regenerating passwords. If credentials do not work, Doctor reports a real
runtime failure instead of hiding it with a fallback.

For JSKIT, this replaces the current hardcoded assumption that a
MariaDB-compatible server exists at `127.0.0.1:3306` with a known root password.
The database endpoint becomes resolved Vibe64 runtime state.

## Doctor After Project Creation

### What The User Sees

Project Doctor should show the runtime as its own set of checks before adapter
checks that depend on it:

```text
Project runtime selection
Project runtime lock
Node 22
MySQL 8.0 service
Runtime config
Generated env files
Dependencies
Verification command
```

If something is missing, the repair action should be explicit:

```text
Realize runtime
Initialize MySQL
Regenerate Env
Install dependencies
Run migrations
```

### What Happens Under The Hood

Doctor validates the source-owned runtime selection against the Vibe64 catalog
and adapter contract. It then verifies that the project runtime root contains
the realized Nix environment and service metadata matching the source lock.

The current setup Doctor checks should move from ambient host checks like:

```text
node --version
npm --version
mysql --version
php --version
composer --version
```

to runtime-aware checks:

```text
resolve Vibe64 runtime env
run node --version inside that env
compare actual version to project runtime lock
report the Nix package/store path
```

This keeps adapters authoritative about technology needs while keeping Vibe64
authoritative about supported versions and fulfillment.

## Running The App

### What The User Sees

The user runs preview, target scripts, Codex workflows, and setup repairs as
usual. There is no container UI and no Docker concept.

Commands just work:

```text
npm install
npx jskit ...
mysql ...
php artisan ...
composer install
```

They work because Vibe64 starts them inside the selected runtime environment.

### What Happens Under The Hood

The terminal/session layer injects the project runtime environment into every
Vibe64-owned command. PATH points at the Nix-realized toolchain before the host
PATH. Runtime config values are injected from the shared Runtime Config engine,
then materialized to generated files only for frameworks that need files.

The source tree remains normal source. The generated app does not need to know
about Nix directly unless Vibe64 deliberately exposes a CLI command for
debugging.

## Deployment

### What The User Sees

Deployment should feel easier than it was with Docker:

1. The project already has a runtime selection and lock.
2. Doctor has already proven the selected runtime in development.
3. Publish uses the same runtime contract.
4. The deployment page shows runtime readiness next to public URL, Env, and
   service status.

The user does not choose a separate production image. They do not create a
production MySQL installation by hand. They do not maintain a Compose file.

### What Happens Under The Hood

Online deployment already has the right broad shape:

```text
create publish source
run prepare/build/migrate
copy release workspace
write environment file
write start script
install/start systemd unit
health check
publish Caddy route
```

The Nix runtime slots into that flow by making every phase runtime-aware:

```text
prepare: run with project runtime env
build: run with project runtime env
migrate: run with project runtime env and production runtime config
serve: start through a Vibe64 runtime wrapper
```

The release service should not depend on ambient Node/PHP/MySQL. Its start
script should enter the same locked Vibe64 runtime before executing the
adapter's serve command.

Production database state is separate from development database state, but the
binary version and service bootstrap logic come from the same project runtime
lock. For one host per customer, Vibe64 can manage the database as a host
service under the customer/owner service root instead of a container volume.

Production values still flow through Runtime Config:

```text
adapter describes required env shape
Vibe64 contributes managed database/auth/deployment values
user contributes editable production values
deployment runner receives resolved env
generated files are materialized only when needed
```

This avoids the current risk of dev Env, deployment Env, and service runtime
being separate systems.

## CLI And Online

This should be a CLI feature first, or at least CLI-equal.

Online should provide the best UI for selecting package versions and viewing
runtime status. The CLI should use the same manifest, catalog, lock, Doctor,
and runtime realization.

Suggested CLI shape:

```text
vibe64 doctor
vibe64 runtime options
vibe64 runtime set node 22
vibe64 runtime set database mysql@8.0
vibe64 runtime realize
vibe64 runtime up
vibe64 runtime status
vibe64 runtime down
```

Online can call the same service methods that back these commands. There should
not be an online-only runtime model.

## Why This Does Not Drift

This design avoids drift because there is one runtime source of truth:

```text
Vibe64 catalog:
  all supported package families and versions

Adapter contract:
  which package families the adapter uses

Project config:
  selected values from the approved catalog

Project runtime lock:
  exact resolved Nix entries

Runtime state:
  realized binaries, service data, credentials, sockets, logs

Doctor:
  verifies actual host/project state against the lock
```

The app does not depend on whatever happens to be installed on the host. The
Online deployment path does not invent a separate runtime. The CLI and UI do
not edit different settings. Generated `.env` files remain output, not truth.

## Detailed Execution Plan

This is the exact work list to make the Nix-backed runtime real. The sequence
matters. Do not deploy this to the VPS until the local vertical slice passes
and the user explicitly asks for deployment.

### 1. Establish Runtime Ownership

- Add a public Vibe64 runtime/toolchain module.
  - Preferred location: `packages/vibe64-runtime/src/server/toolchain/`.
  - It should own catalog loading, adapter requirement resolution, lock
    generation, Nix command planning, and runtime env resolution.
  - Do not put Nix behavior directly inside adapters.
  - Do not put project-specific runtime state inside `studio-terminal-core`.

- Define the runtime ownership contract in code:
  - Vibe64 catalog owns supported package families and versions.
  - Adapter contract owns which package families a project type needs.
  - Project config owns selected choices from the approved catalog.
  - Runtime lock owns exact resolved Nix inputs/packages.
  - Runtime state owns realized binaries, service data, credentials, sockets,
    logs, and process metadata.

- Add tests for the ownership contract:
  - Adapter requests unsupported package family -> hard failure.
  - Project selects unsupported version -> hard failure.
  - Missing runtime lock -> Doctor reports repairable runtime realization.
  - Runtime lock mismatch -> Doctor reports hard failure unless explicitly
    regenerated.

### 2. Add The Vibe64 Catalog

- Add a versioned catalog file or module in the public repo.
  - Suggested path: `packages/vibe64-runtime/src/server/toolchain/catalog.js`.
  - It must be deterministic and sorted.
  - It must not read host state to decide supported versions.

- Initial base catalog:
  - `git`
  - `ripgrep`
  - `playwright`
  - `codex`
  - `opencode`
  - `bubblewrap`
  - `node@22`

- Initial project catalog:
  - `node@22`
  - `mysql@8.0`

- Add later catalog entries only after the JSKIT slice works:
  - `node@20`
  - `mysql@8.4`
  - `postgres@16`
  - `postgres@17`
  - `php@8.3`
  - `php@8.4`
  - `composer@2`
  - `laravel-installer`

- Each catalog entry must contain:
  - stable package id
  - display label
  - package family
  - Vibe64-supported version
  - Nix package expression or flake attribute
  - command names provided by the package
  - Doctor version command
  - Doctor version parser
  - whether it is a base tool, project tool, or service runtime

- Add catalog tests:
  - catalog ids are unique
  - family/version pairs are unique
  - command names are deterministic
  - default versions exist in allowed versions
  - every enabled catalog entry has a Doctor check definition

### 3. Add Nix Host Checks

- Replace ambient Studio setup checks for base tools with Nix-aware checks.
  - Current location to refactor:
    `packages/studio-setup-doctor/src/server/service.js`.

- Doctor should check:
  - Linux host
  - Nix command exists
  - Nix can evaluate the Vibe64 base toolchain
  - Nix can realize the Vibe64 base toolchain
  - base tool versions match the Vibe64 catalog
  - `bwrap` binary exists and host capabilities allow it to run
  - Playwright can launch Chromium
  - Codex/opencode command exists through the runtime and auth state is usable

- Add a repair action:
  - If Nix is missing, show a host setup command.
  - If the base toolchain is unrealized, run a Vibe64-controlled realization
    command.
  - Do not silently install Nix.
  - Do not silently fall back to host binaries.

- Add tests:
  - Nix missing -> hard stop with repair.
  - Nix present but realization fails -> hard stop.
  - Realized tool version mismatch -> hard stop.
  - Capability failure for `bwrap` -> hard stop with host-capability message.
  - Playwright binary present but launch fails -> hard stop with browser smoke
    output.

### 4. Define Project Runtime Manifest And Lock

- Add source-owned runtime selection storage.
  - Keep adapter config under existing source-owned config:
    `<project>/.vibe64/config/*`.
  - Add exact resolved runtime lock:
    `<project>/.vibe64/runtime.lock.json`.

- Runtime selections should be written through the same project config flow that
  currently writes `.vibe64/project_type` and `.vibe64/config/*`.

- Runtime lock must include:
  - schema name and version
  - Vibe64 catalog version
  - adapter id
  - project type
  - selected tools
  - selected services
  - resolved Nix flake/input identity
  - resolved Nix package attributes
  - expected command versions
  - lock creation timestamp

- Runtime lock must not include:
  - passwords
  - host paths
  - sockets
  - ports
  - PID files
  - machine-specific service state

- Add lock tests:
  - stable key ordering
  - no secrets
  - no absolute host paths
  - exact regeneration is deterministic for the same inputs
  - changing catalog version changes lock identity

### 5. Add Adapter Runtime Contracts

- Add a method to adapter definitions:
  - `getRuntimeRequirements(context)`

- JSKIT initial contract:
  - requires `node@22`
  - requires package manager support through Node/Corepack/npm
  - if `jskit_database_runtime=mysql`, requires `mysql@8.0` service
  - if `jskit_database_runtime=none`, requires no database service
  - if `jskit_database_runtime=postgres`, report unsupported service
    orchestration until Postgres is implemented

- Laravel later contract:
  - requires PHP
  - requires Composer
  - requires Node
  - optionally requires MySQL/Postgres/SQLite based on adapter config

- Next.js later contract:
  - requires Node
  - optionally requires MySQL/Postgres based on adapter config

- Add tests:
  - JSKIT none database resolves only Node.
  - JSKIT MySQL resolves Node and MySQL.
  - JSKIT Postgres reports unsupported until implemented.
  - Adapter cannot request packages outside Vibe64 catalog.
  - Adapter cannot force an unapproved version.

### 6. Build Runtime Env Resolution

- Add a runtime env resolver:
  - input: target root, project config, adapter contract, runtime lock

### 11. Preserve Adapter-Owned Hosted Config

`docs/adapter-owned-hosted-config-plan.md` is pre-existing baseline for this
work, not new Nix scope. The Nix pass must preserve that boundary while adding
runtime/package determinism.

- Keep the ownership split intact:
  - Vibe64 owns supported runtime versions, Nix resolution, hosted service
    roots, generic secret storage, and generic Runtime Config plumbing.
  - Adapters own hosted config schema, settings sections/actions, readiness
    messages, seeding behavior, and the app-specific meaning of generated Env
    or files.

- Do not add generic Vibe64 app config forms for adapter-specific choices.
  Runtime UI can select supported packages and service versions. JSKIT, Laravel,
  and future adapters still decide what MySQL, Supabase, Composer, Artisan, or
  app auth mean to that project type.

- Integrate Nix-managed services through adapter-owned config paths:
  - Vibe64 may provide pinned MySQL from the Nix catalog.
  - JSKIT decides whether the target uses the MySQL-compatible runtime.
  - JSKIT maps the project database and materializes JSKIT app env/config.
  - Laravel decides whether its app config needs MySQL and how migrations/env
    should be run.
  - Runtime Config remains the bridge from managed values to generated output.

- Add regression checks for the boundary:
  - Vibe64 runtime lock contains package/service identity but no Supabase,
    Laravel, JSKIT auth, or app-specific settings semantics.
  - Managed MySQL credentials flow into JSKIT Runtime Config through the JSKIT
    adapter, not through a generic Vibe64 MySQL form.
  - Laravel database env and migration commands are produced by the Laravel
    adapter, not by shared Vibe64 app-config logic.
  - Non-JSKIT projects never see Supabase requirements.
  - Supabase remains adapter-owned config and adapter-owned secret handling.
  - Generated `.env` files remain compatibility output, not runtime truth.

### 19. Done Criteria

The Nix runtime work is done when:

- Vibe64 can start on a clean Linux host with Nix and no Docker.
- Studio Doctor verifies Nix-backed base tools.
- JSKIT project setup can select Node 22 and MySQL 8.0.
- Project runtime lock is committed source-owned config.
- Runtime state and secrets stay out of source.
- Doctor verifies project runtime lock and realized binaries.
- MySQL initializes with app DB/user/password automatically.
- Runtime Config gets DB values from managed service state.
- Generated `.env` files are synced from Runtime Config.
- Preview, target scripts, Doctor repairs, and workflow commands use the same
  runtime env.
- Online publish uses the same runtime env.
- Release services do not depend on ambient host Node/PHP/MySQL.
- CLI and Online read/write the same runtime state.
- Adapter-owned hosted config is in place for JSKIT auth/database settings.
- Supabase is optional JSKIT adapter config, not Vibe64-level setup.
- No Docker/container runtime code remains.
- No ambient host package check is treated as success for project runtime.
- No compatibility shim or silent fallback is required.
