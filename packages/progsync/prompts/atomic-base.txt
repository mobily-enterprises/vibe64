PROGRAM FORMAT

- Exactly one H1 title for the complete file.
- An optional short preamble.
- Exactly one `## Uses` section.
- Exactly one `## Provides` section.
- When the module uses nothing outside itself, Uses contains exactly
  `- Nothing outside this file.` It is never left empty.
- Every Uses item is an exact Markdown link to an operational dependency.
- Every Uses link has a backticked visible symbol and a canonical provider:
  `@/<program-path>.md#<anchor>`, `package:npm/<package>#<anchor>`,
  `platform:<capability>#<anchor>`, or `asset:<identity>`.
- Project-owned links begin at `@/`, meaning the `program/` root. Never use
  `../` links.
- Standalone exported functions use their exact exported name followed by
  parentheses in level-three backticked headings beneath Provides, such as
  `### `dispatchAlerts()``. Exported non-callable symbols use their exact name.
- Every exported class uses this exact structure, replacing `ClassName` and its
  lowercase Markdown anchor with the real exact identity:

  ```markdown
  ## Provides

  - The exported class [`ClassName`](#class-classname).

  ## Class `ClassName`
  ```

  Place intentionally public methods beneath its Class section as level-three
  headings ending in parentheses.
- The first sentence beneath an operation heading begins with `The function`
  or `The method`, then states every exact parameter name and its semantic type,
  and its return type or that it returns no value.
- That complete opening signature is one sentence and contains the word
  `returns` before its first full stop. Put behavior in later sentences.
- Do not label an operation synchronous or asynchronous. Promise and `async`
  mechanics belong to the target implementation. State ordering, concurrency,
  or completion requirements only when they are part of the program's meaning.
- Private helpers, temporary variables, internal data structures, and target
  syntax do not receive headings. Fold their meaningful behavior into the
  public symbol that uses them.
- Describe exact calls only when they cross the module boundary. Do not expose
  a call to another same-file function merely because that function is also
  exported; state the local transformation or behavior instead.
- State exact public inputs, results, local selection and transformation,
  meaningful conditions, ordering, repetition, mutation, outside calls and
  arguments, used results, effects, failures, and important reasons.
- Every meaningful value must come from a named input, an exact field of a
  declared complex type, a previous result, a stated literal, module-owned
  state, or an exact outside dependency. Express field ownership in readable,
  language-neutral terms rather than target member-access syntax.
- Use a numbered list when an orchestration function has several meaningful
  ordered transformations. Each point follows semantic data from its source to
  its use; it must not expose private helpers or target mechanics.
- Do not produce line-by-line pseudocode.
- Shared complex object shapes use reachable definitions from `types.md`.
  Primitive values and ordinary identifiers do not become shared types merely
  because a variable contains them.
- When the implementation defines a previously unregistered complex shape that
  crosses the public boundary, add its concise definition to
  `program/types.md` and refer to it as `[Type name]` in Program prose. Do not
  repeat its fields in the module or create types for private temporary objects.
- `[Type name]` is the complete type-reference syntax. Every such name resolves
  implicitly and case-sensitively through `program/types.md`. Types never appear
  in Uses and never carry provider paths in authored Program.
- A new type is a level-three backticked heading beneath the one `## Provides`
  section in `program/types.md`, followed by its concise field meanings. Keep
  the file's one `## Uses` section; use `- Nothing outside this file.` when it
  has no dependencies.

MODE RULES

CREATE_PROGRAM:
Create the complete Program file from the implementation. Describe only its
public semantic module. Create missing shared complex types when their exact
shape is established by the implementation. Use mechanically supplied provider
identities. If an important origin, type, or behavior cannot be established,
return blocked instead of guessing.

CREATE_IMPLEMENTATION:
Create a complete target file from Program, resolved definitions, target
bindings, and retained context. Choose private realization conservatively. Do
not invent undeclared public meaning.

PROGRAM_TO_IMPLEMENTATION:
Compare previous and current Program, then make the smallest implementation
patch that realizes all current Program. Preserve compatible implementation
changes and unrelated realization details.

IMPLEMENTATION_TO_PROGRAM:
Separate realization-only implementation work from changed public meaning.
Leave Program unchanged for private refactoring, optimization, formatting,
comments, compatible framework changes, and visual tuning. When public
behavior, data flow, outside calls, effects, meaningful failures, constraints,
or important reasons changed, make the smallest corresponding Program patch.
Add or refine a shared complex type only when that changed public boundary
requires it.

An implementation repair that makes behavior conform more accurately to
meaning already stated by Program is realization-only. For example,
floating-point compensation used to implement an already-declared rounding
rule remains solely in implementation; do not add the compensation mechanism
to Program.

RECONCILE_BOTH:
Preserve compatible work from both current files. If the implementation
already realizes a new Program statement, keep it. If the files make materially
incompatible public claims, return blocked and leave both unchanged.

NO_CHANGE:
Write nothing.

PRESERVATION

Unless Program requires otherwise, preserve public behavior, private helpers,

layout, responsive behavior, transitions, accessibility, test hooks, framework
conventions, error mechanics, and compatible manual refinements. Correctness
