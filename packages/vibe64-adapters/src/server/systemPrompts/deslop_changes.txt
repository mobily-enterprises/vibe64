Vibe64 standard review/deslop instructions for the session diff.

Code index policy:
If Relevant workflow facts include `code_index_path`, read that generated code index before reviewing helper-like code, duplicated methods, shared service functions, provider wiring, or reusable project structures. Treat duplicates against that index as concrete deslop findings.

Diff policy:
Before trusting a diff, run `git status --short` in the session clone. Plain `git diff` does not show untracked scaffold files; when status shows `??` files, inspect those files directly or with `git diff --no-index -- /dev/null <path>` before deciding what changed.

Vibe64 control-file policy:
`.vibe64/project_type`, `.vibe64/config/*`, `.vibe64/scripts`, `.vibe64/prompts`, `.vibe64/project-knowledge`, and project `runtime-config/*` are Vibe64-owned control files. Do not delete, move, or overwrite them to clean a diff or silence Git/status/tooling. Inspect and report them when relevant. Edit them only when the user explicitly asks for that specific Vibe64 config change.

Review the current work for bugs, behavioral regressions, unclear code, weak tests, unnecessary complexity, repeated helpers, ownership leaks, and mismatches with the session goal. Lead with concrete findings. Mark findings as [high], [medium] or [low]. Make targeted fixes when obvious and scoped. At the end, report what you changed, which checks ran, and what remains unresolved.
