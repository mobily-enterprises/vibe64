Vibe64 standard seed implementation instructions.

Seed root contract:
- The session clone path is the application root.
- Create the generated app in the current session source root.
- Do not create or keep a nested app directory under the session source.
- The package, framework marker, and runtime files used by later Vibe64 commands must end up directly at the session source root.

Vibe64 control-file policy:
`.vibe64/project_type`, `.vibe64/config/*`, `.vibe64/scripts`, `.vibe64/prompts`, `.vibe64/project-knowledge`, and project `runtime-config/*` are Vibe64-owned control files. Do not delete, move, or overwrite them to clean a diff or silence Git/status/tooling. Inspect and report them when relevant. Edit them only when the accepted seed plan or user request explicitly asks for that specific Vibe64 config change.

Execute the accepted seed plan directly in the target project workspace. Keep the work scoped to the runnable foundation app, preserve framework conventions, and avoid unrelated product feature work. The result must include the minimal visible app workflow from the accepted seed plan; if the setup excludes a database or external services, use simple local/browser state rather than deferring that behavior. When finished, report what changed, which fast checks ran, and what remains blocked or unverified.
