Sync code with local repo

Vibe64 standard local-source sync instructions for a session clone.

Goal:
Make this session clone and the opened local repository agree when that is safe. The opened repository is the user's canonical source. It may itself have an upstream remote, but this action must not assume GitHub and must not use GitHub issue, pull request, fork, or merge workflows unless the user explicitly asks for them.

Important boundary:
This action may commit ordinary source-owned session work before syncing it back to the opened local repository. It must not inspect provider credentials, log in to GitHub, force-push, discard work, chmod home directories, or commit ambiguous ownership. This is a Git/source sync action, not a full app verification action.

Ownership rules for dirty files:
- Safe source-owned project changes may be committed. This includes normal application source files and portable Vibe64 project contract files such as `vibe64.project.json`, `vibe64.runtime-lock.json`, optional `.vibe64/scripts`, `.vibe64/prompts`, and `.vibe64/project-knowledge`.
- Do not commit Vibe64-owned runtime/session/deployment/credential state, secrets, local environment files, runtime config values, git cache data, deployment state, or account/user metadata.
- Treat unclear ownership, unrelated files, unresolved conflicts, suspected secrets, and inaccessible opened-repository paths as blockers.

Fast path first:
- Start with `git status --short --branch` in the session source.
- If the session working tree is dirty, inspect only enough to classify ownership. Commit safe source changes with a clear message before syncing. If any blocker remains, stop before touching the opened repository and report it.
- Identify the session `HEAD`, the opened repository path from Vibe64 session facts, and the opened repository `HEAD`.
- If the session `HEAD` already equals the opened repository `HEAD`, stop and report that the local repo is already synced.
- If the opened repository `HEAD` is an ancestor of the session `HEAD`, update the opened repository by a normal fast-forward or explicit local push into the opened repository branch. Verify the opened repository now points at the session `HEAD`.
- If the session is behind or diverged from the opened repository, fetch from the opened repository, merge normally into the session branch when safe, resolve only clear source conflicts, and then sync back. Do not rebase unless the user explicitly requested rebase in this same turn.

Verification and report:
- Run focused non-interactive source checks only when this turn changes files, creates commits, resolves conflicts, or when the user explicitly asks for verification.
- Do not run broad verification, UI/runtime verification, preview restarts, dev-server restarts, login flows, or auth-user preparation from this sync action.
- Always report in plain language first, then include a collapsed `<details>` section with starting and final session/opened-repo commits, commands run, files committed, checks run, and blockers.
