Sync code with GitHub

Vibe64 standard Git sync instructions for a session clone.

Goal:
Make this session clone and the configured GitHub remote default branch agree when that is safe. Treat the session branch and the remote default branch as the only user-facing Git states unless the user explicitly asks for a named branch or pull request workflow. This is the user-facing unified sync action: commit safe source changes, receive safe remote changes, and push safe local commits.

Important boundary:
This action is allowed to commit ordinary source-owned session work before syncing. It is not allowed to stash, discard, reset away, force-push, or silently commit ambiguous ownership. If the working tree has uncommitted changes, inspect and classify them before any commit.
This is a Git/source sync action, not a full app verification action. Do not start, stop, restart, or reconfigure app servers, preview servers, managed previews, browser sessions, auth users, cookies, or runtime services unless the user explicitly asked for that exact verification in this same turn.

Managed Git command path:
- Use `git` and `gh` exactly as provided by the Vibe64 session environment.
- Do not call absolute or host Git/GitHub binaries such as `/usr/bin/git`, `/bin/git`, `/usr/local/bin/git`, `/usr/bin/gh`, or `/bin/gh`.
- Do not bypass the managed command path with `command -p`, stripped `PATH`, `env -i`, shell builtins that force host lookup, or direct executable paths.
- If managed `git` or `gh` cannot authenticate or cannot reach the remote, report that managed-command failure. Do not retry with host binaries, inspect credentials, or invent alternate authentication.

Ownership rules for dirty files:
- Safe source-owned project changes may be committed. This includes normal application source files and portable Vibe64 project contract files such as `vibe64.project.json`, `vibe64.runtime-lock.json`, `vibe64.system.json`, optional `.vibe64/scripts`, `.vibe64/prompts`, and `.vibe64/project-knowledge`.
- Expected generated source artifacts may be committed only when they are part of the project source contract and do not contain secrets or environment-local state. Mention them in the report.
- Do not commit Vibe64-owned runtime/session/deployment/credential state, secrets, local environment files, runtime config values, git cache data, deployment state, or account/user metadata.
- Treat source-tree `.vibe64` additions outside optional `.vibe64/scripts`, `.vibe64/prompts`, and `.vibe64/project-knowledge` as ownership drift, not as ordinary sync changes. Stop and explain the blocker unless the user explicitly asked this same turn to handle that Vibe64 source contract change.
- Treat unclear ownership, unrelated files, unresolved conflicts, and suspected secrets as blockers. Explain the exact paths and why sync stopped.

Fast path first:
- Start with `git status --short --branch`.
- If the working tree is dirty, inspect tracked and untracked changes only enough to classify ownership. Commit safe source changes with a clear message before syncing. If any blocker remains, stop before fetching/pushing and report it.
- If the working tree is clean after any required commit, identify the current branch/`HEAD`, the configured remote, and the live remote default branch with the fewest commands that answer those questions. Prefer `git ls-remote --symref origin HEAD` for the live default branch over stale local assumptions.
- Fetch only the relevant remote default branch before comparing history.
- Compare `HEAD` with the fetched remote default branch using ancestry checks, not commit messages.
- If `HEAD` equals the fetched remote default branch, stop and report that the session is already synced with GitHub.
- If `HEAD` is an ancestor of the fetched remote default branch, fast-forward the session branch to the fetched remote default branch and stop.
- If the fetched remote default branch is an ancestor of `HEAD`, push with an explicit refspec from the current `HEAD` to the remote default branch. Then verify the live remote ref points at `HEAD` and stop.
- If neither side is an ancestor of the other, this is normal divergent sync work, not a blocker. Follow the Divergence policy below.
- In these fast paths, do not run broad rediscovery, repeated status/diff passes, or unrelated tests. Keep the report short: starting `HEAD`, final `HEAD`, remote branch, files committed if any, whether the action was already-current, fast-forward, or push, and whether checks were skipped because this was a Git-only fast path.

Sync policy:
- If there is no configured remote, the remote is not GitHub, GitHub authentication is unavailable, or the remote cannot be reached, report the exact blocker. Do not try to log in, inspect secrets, or invent credentials.
- If dirty changes are all safe source-owned changes, commit them with a concise message before remote sync. Do not ask the user to use a separate commit workflow for ordinary source edits.
- If the session and remote default branch have diverged, do not stop merely to ask whether to merge or rebase. The default unified-sync strategy is a normal non-force merge of the fetched remote default branch into the current session branch, because it preserves both histories and keeps the session branch recoverable.
- For divergent sync, merge the fetched remote default branch into the session branch with a normal merge commit. Do not rebase unless the user explicitly requested rebase in this same turn. If the merge succeeds, run relevant checks when files changed, then push with an explicit refspec from the current `HEAD` to the remote default branch and verify the remote ref.
- If the merge has conflicts, inspect the conflicts and resolve only clear, scoped source conflicts. After resolving, commit the merge, run relevant checks, push with an explicit refspec, and verify the remote ref.
- A conflict is clear only when the resolution is mechanical and does not choose product behavior for the user. Route/API output contracts, service behavior, data schemas, auth, deployment, runtime-config ownership, generated metadata, migrations, and test expectation conflicts are not clear merely because the files can be made to compile; ask a concrete decision question unless one side is purely formatting or a generated counterpart to an already-decided source change.
- If conflicts, ownership, generated artifacts, or secrets are ambiguous, abort the merge to restore the pre-merge session branch, then ask the user a concrete decision question before continuing. The question must name the conflicting paths, summarize the competing changes from the session branch and remote default branch, and offer the safest specific options. Do not ask a generic merge-vs-rebase question, and do not leave the repository in a conflicted merge state unless the user explicitly asked to continue conflict resolution manually.
- If the remote default branch changes while syncing, re-check ancestry before any push. Push only when the fetched remote default branch is still an ancestor of `HEAD`.
- If a push is rejected because the remote default branch moved, fetch the new remote default branch and repeat the safe ancestry/divergence flow once. Stop if the second attempt still cannot be completed without ambiguous conflict or ownership decisions.
- Never force-push the remote default branch. Never reset away local work. Never delete local or remote branches unless the user explicitly asks for that exact branch operation.
- If the user explicitly asks to sync a named branch instead of the default branch, use explicit refspecs and report the branch name.

Verification and report:
- Run focused non-interactive source checks only when this turn changes files, creates commits, resolves conflicts, or when the user explicitly asks for verification. Prefer syntax checks, targeted unit tests, or focused command-line checks for the touched files. Do not repeat expensive checks when the session was already current, moved by a clean fast-forward, or pushed already-verified commits. If checks are skipped after committing, explain why.
- Do not run broad verification or UI/runtime verification from this sync action. Do not run `npm run verify`, `npx jskit app verify-ui`, Playwright/browser tests, UI receipt refreshes, preview restarts, dev-server restarts, login flows, cookie/profile mutation, or any command that can change the user's active preview/browser/auth state unless the user explicitly asked for that verification in this same turn.
- If UI-visible files changed and a JSKIT UI receipt, browser check, or preview restart would normally be needed, do not perform it from Sync code with GitHub. Report it as remaining verification for a separate explicit UI-check action.
- After any push, verify the live remote ref points at the intended commit.
- Always report in two parts. Start with one or two plain-language sentences saying what happened and whether the session and GitHub are now synced.
- Then put the technical explanation in one collapsed section exactly shaped as:
  `<details>`
  `<summary>Technical details</summary>`
  starting and final `HEAD`, the remote branch and commit used, whether the session is now equal/ahead/behind/diverged, commands run, files committed, checks run, and any unresolved conflicts or blockers
  `</details>`
- Put each details tag on its own line. Never combine `<details>` and `<summary>` on one line.
