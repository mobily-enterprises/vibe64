Sync with remote

Vibe64 standard Git sync instructions for a session clone.

Goal:
Bring this session clone up to date with the project remote while preserving the user's session work. Treat the session branch and the remote default branch as the only user-facing Git states.

Fast path first:
- Start with `git status --short --branch`.
- If the working tree is clean, identify the current branch/`HEAD`, the configured remote, and the live remote default branch with the fewest commands that answer those questions. Prefer `git ls-remote --symref origin HEAD` for the live default branch over stale local assumptions.
- Fetch only the relevant remote default branch before comparing history.
- Compare `HEAD` with the fetched remote default branch using ancestry checks, not commit messages.
- If `HEAD` already equals the fetched remote default branch, stop and report that the session is already current.
- If `HEAD` is an ancestor of the fetched remote default branch and there are no local-only commits, fast-forward the session branch to the fetched remote default branch and stop.
- In this fast path, do not run broad rediscovery, repeated diff/status passes, or unrelated tests. Keep the report short: starting `HEAD`, final `HEAD`, remote branch, and whether any files changed.

Sync policy:
- If there is no configured remote or the remote cannot be reached, report the blocker clearly.
- If the working tree has uncommitted changes, inspect them, preserve them, and only stash or commit when that is the smallest safe way to complete the sync. Ask the user if ownership is unclear.
- If the session has local commits and the remote default branch has moved, merge or rebase the remote default branch into the session only when it is clean and safe. Prefer the repository's existing Git style if obvious; otherwise use the smallest history-preserving operation and explain it.
- Never force-push, reset away local work, delete branches, or rewrite published history unless the user explicitly requests that exact operation.

Verification and report:
- Run focused checks only when the sync applies file changes that affect behavior, when conflict resolution changes files, or when the user explicitly asks for verification. Do not repeat expensive checks when the session was already current or moved by a clean fast-forward with no local conflict.
- Always report in two parts. Start with one or two plain-language sentences saying what happened and whether the session is now up to date with the remote.
- Then put the technical explanation in one collapsed section exactly shaped as:
  `<details>`
  `<summary>Technical details</summary>`
  starting and final `HEAD`, the remote branch and commit used, whether the session is now ahead/behind/diverged, commands run, and any unresolved conflicts or blockers
  `</details>`
- Put each details tag on its own line. Never combine `<details>` and `<summary>` on one line.
